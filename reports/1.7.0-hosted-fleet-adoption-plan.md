# Beam 1.7.0 Hosted Fleet Adoption Plan

## Summary

Beam `1.7.0` is the release that turns the OpenClaw fleet layer from an internal production system into something another operator can adopt without repo archaeology.

The goal is not more protocol breadth. The goal is to make Beam installable, operable, and supportable by people who did not build it.

Release intent:

- central Beam remains the single identity, policy, and audit authority
- OpenClaw remains an external connector and host daemon, not a patched runtime
- Beam IDs remain the only identity model
- destructive fleet actions stay explicitly gated
- adoption means fresh-machine onboarding, shared operator administration, visual stability, and supportable incidents

## Release Theme

`1.7.0` is the hosted-fleet-adoption release:

- packaged host install and bootstrap tokens
- guided enrollment and operator onboarding from the UI
- real operator/member management on top of RBAC
- visual regression baselines for the dashboard
- fleet analytics and support-bundle export
- external dogfood evidence across multiple operators and hosts

## Why This Is Next

`1.6.0` made Beam boring enough to claim production-readiness for a controlled fleet:

- external alerting works
- role boundaries and destructive guards are real
- backup, restore, recovery, and rollback drills are evidenced
- multi-day soak evidence exists
- release gating is evidence-backed

What still keeps Beam too insider-heavy for broader adoption:

- host setup still assumes repo access and local shell comfort
- enrollment is safe, but not yet packaged as a truly guided product flow
- operator administration exists as permission logic, not yet as a full user-management surface
- the improved UI has a smoke gate, but not a baseline diff discipline
- support and incident exports still require too much manual reconstruction
- production proof is still mostly internal; the next step is external operator adoption

`1.7.0` fixes exactly that.

## P0 Before 1.7.0

- packaged host installer with bootstrap token flow
- guided host enrollment and operator onboarding in the UI
- operator/member management and role assignment surface
- screenshot baseline and visual regression gate for the dashboard
- fleet analytics and support bundle export
- external dogfood evidence across multiple operators and hosts
- fresh installer/buyer/operator evidence
- release gates that treat adoption readiness as a hard cut condition

## P1 After 1.7.0

- hosted billing and commercial account controls
- richer historical fleet analytics and SLA reporting
- more runtime connectors beyond OpenClaw
- automated canary promotion policies

## Explicit Non-Goals

- no free-form custom DID editor
- no OpenClaw core patching
- no generic connector marketplace
- no Slack-style collaboration product
- no protocol-family headline change

## Workstreams

### `#191` Package the host installer and bootstrap token flow

Ship a truly operator-friendly host install path.

Capabilities:

- packaged macOS and Linux host install path
- bootstrap token minted in Beam for first join
- one-command enroll path for a fresh machine
- unified install / uninstall / status semantics for the host daemon

Exit criteria:

- a fresh machine can join a Beam fleet with one guided command
- the bootstrap token is short-lived and auditable
- install failures are clear without reading repo source

### `#192` Add guided enrollment links and operator onboarding

Move the host-join path into the product surface.

Capabilities:

- generate enrollment links and copyable install commands in the UI
- show token expiry, approval state, and last-seen state during enrollment
- add a guided “join a new host” operator path for macOS and Linux

Exit criteria:

- an operator can onboard a new host from the dashboard without local code edits
- enrollment and approval state stay visible from start to finish
- the docs mirror the same product path

### `#193` Add operator/member management and RBAC administration

Turn RBAC from hidden behavior into explicit operator administration.

Capabilities:

- invite users
- assign `viewer`, `operator`, and `admin`
- audit role changes
- expose current role posture in the dashboard

Exit criteria:

- one admin can add another operator without repo changes
- role changes are visible and auditable
- risky actions still honor the same destructive guards

### `#194` Add screenshot baselines and visual regression gates

Make the dashboard look stable, not just functional.

Capabilities:

- baseline screenshots for:
  - login
  - fleet
  - workspace
  - intents
  - trace
- CI diff artifacts when visuals move
- release visibility for unexpected UI drift

Exit criteria:

- the UI gate compares against a committed baseline
- regressions produce readable diff artifacts
- release confidence is not based on human memory alone

### `#195` Add fleet analytics and support-bundle export

Make incidents supportable without reconstructing the story by hand.

Capabilities:

- latency and delivery trend summaries
- host uptime and route churn summaries
- conflict frequency and remediation trend summaries
- support-bundle export for:
  - host
  - workspace
  - trace
  - recent fleet digest / alert history

Exit criteria:

- one real incident can be exported as a support bundle
- an operator can hand a concise bundle to another operator without replaying logs manually
- the dashboard exposes enough history to explain the incident path

### `#196` Collect external dogfood and adoption evidence

Prove Beam works for more than its author.

Scope:

- at least two human operators
- at least five hosts
- at least one fresh-machine onboarding outside the original development machine
- at least one real support-bundle export and one recovery follow-up

Exit criteria:

- external adoption evidence is committed
- discovered blockers become GitHub issues, not tribal memory
- the path is credible for a non-author operator

### `#197` Run 1.7.0 installer/buyer/operator dry runs

Commit fresh adoption evidence for the release claim.

Required reports:

- `reports/1.7.0-installer-dry-run.md`
- `reports/1.7.0-buyer-dry-run.md`
- `reports/1.7.0-operator-dry-run.md`

Coverage:

- fresh-machine install and enrollment
- buyer-style host add / approval / first message path
- operator role management, support bundle, and incident handling

Exit criteria:

- the install path can be followed without repo archaeology
- the operator path is explicit and timed
- all discovered blockers are filed as real issues

### `#198` Prepare 1.7.0 cut checklist, release notes, and go/no-go gates

Only ship when adoption readiness is fully evidenced.

Required proof:

- packaged host install works on a fresh machine
- guided enrollment works from the dashboard
- operator/member administration works
- visual regression gate runs in CI
- support-bundle export works
- external dogfood evidence is committed
- installer/buyer/operator reports are committed
- final release smoke proves API, docs, public status, npm, and onboarding surface

## Acceptance Criteria

- a fresh macOS or Linux machine can join Beam with one guided install flow
- at least two human operators can administer the same fleet
- visual regressions produce committed baselines and CI diff artifacts
- one real incident can be exported as a support bundle
- external dogfood evidence exists across five hosts and two operators
- final release smoke proves Beam `1.7.0` across API, docs, public status, npm, and hosted fleet onboarding
