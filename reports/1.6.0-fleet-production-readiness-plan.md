# Beam 1.6.0 Fleet Production Readiness Plan

## Summary

Beam `1.6.0` is the release that turns the OpenClaw fleet from a strong operator console into a boring production system.

The goal is not more protocol surface. The goal is to remove the remaining reasons an operator still needs too much repo knowledge, too much manual supervision, or too much trust in luck.

Release intent:

- central Beam remains the single identity, policy, and audit authority
- OpenClaw remains an external connector and host daemon, not a patched runtime
- Beam IDs remain the only identity model
- destructive fleet actions stay explicitly gated
- production ready means safe day-2 operations, not more feature breadth

## Release Theme

`1.6.0` is the fleet-production-readiness release:

- external alerting for fleet exceptions
- real RBAC and destructive-action guards
- backup, restore, and recovery drills
- connector upgrade and rollback discipline
- multi-day soak evidence across real hosts
- explicit go/no-go gates for the production cut

## Why This Is Next

`1.5.0` completed the autonomy layer:

- maintenance mode, drain, and resume
- rollout rings and canary visibility
- policy packs and workspace templates
- guided remediation and reconciliation
- recurring rollout and maintenance digests
- local onboarding, UI proof, and CI dashboard evidence

What is still not boring enough for a true production claim:

- important exceptions are still too UI-local
- the admin surface is still too coarse for destructive actions
- host loss and control-plane recovery need a harder drill path
- upgrade safety still relies too much on human memory and sequence discipline
- long-running fleet stability is proven in drills, not yet in a true soak lane

`1.6.0` fixes exactly that.

## P0 Before 1.6.0

- external alert delivery and escalation loop
- RBAC plus destructive-action approval guards
- backup, restore, and host recovery drill
- connector upgrade and rollback runbook with visible state
- multi-day fleet soak evidence
- fresh buyer/operator/incident evidence
- release gates that treat production-readiness as a hard cut condition

## P1 After 1.6.0

- richer fleet analytics and historical trends
- screenshot diff baselines for dashboard UI smoke
- more opinionated host installer packaging
- more runtime connectors beyond OpenClaw

## Explicit Non-Goals

- no free-form custom DID editor
- no OpenClaw core patching
- no generic connector marketplace
- no Slack-style collaboration product
- no new protocol-family headline feature

## Workstreams

### `#181` Add external fleet alerting and escalation delivery

Ship real outbound alert delivery for fleet exceptions.

Capabilities:

- external delivery targets for fleet alerts
- Slack/email-ready alert messages for:
  - stale hosts
  - failed deliveries
  - duplicate conflicts
  - revoked or draining hosts blocking traffic
  - reconciliation failures
- delivery audit trail and retry state
- operator-visible “last delivered / failed / muted” state

Exit criteria:

- a real fleet exception can leave the UI and reach an external destination
- delivery success or failure is visible in Beam
- major fleet exceptions are no longer only visible by opening the dashboard

### `#182` Add RBAC and destructive-action guards

Split the operator surface into real roles.

Roles:

- `viewer`
- `operator`
- `admin`

Guarded actions:

- host revoke
- credential rotate/recover
- conflict resolution and route owner changes
- drain/resume/maintenance transitions
- template apply and destructive remediation

Exit criteria:

- read-only viewers cannot mutate fleet state
- operators can do normal runtime operations
- only admins can perform the highest-risk actions
- destructive actions require explicit confirmation and are auditable

### `#183` Add backup, restore, and recovery drill coverage

Make central Beam failure and host loss survivable in a boring way.

Capabilities:

- documented backup artifact path for the control-plane database
- restore flow for a fresh Beam node
- host recovery path on a fresh machine using existing workspace and fleet state
- repo-owned recovery drill script and evidence

Exit criteria:

- a restored Beam instance can serve fleet state correctly
- a recovered host can rejoin without rebuilding workspace identity state from scratch
- the recovery path is documented and repeatable

### `#184` Add connector upgrade and rollback discipline

Treat connector rollout as a production operation, not just visibility.

Capabilities:

- desired connector version by rollout ring
- explicit rollback target
- operator-visible ring lag and rollback state
- upgrade and rollback runbook embedded into the fleet surface
- rollback evidence in audit and digest output

Exit criteria:

- one canary ring can move forward and back with visible state
- the fleet UI shows who is ahead, behind, or rolled back
- rollback stops being an undocumented manual sequence

### `#185` Add multi-day fleet soak evidence

Prove boringness over time, not just in one drill.

Scope:

- at least three hosts
- recurring heartbeats and inventory
- real inbound/outbound traffic
- at least one maintenance event
- at least one connector rollout or recovery event

Exit criteria:

- a 7-day soak report exists
- a 14-day continuation lane is defined if the 7-day soak exposes real gaps
- all discovered blockers become GitHub issues, not implicit memory

### `#186` Run 1.6.0 buyer/operator/incident dry runs

Commit fresh evidence for the production-ready claim.

Required reports:

- `reports/1.6.0-buyer-dry-run.md`
- `reports/1.6.0-operator-dry-run.md`
- `reports/1.6.0-incident-drill.md`

Coverage:

- buyer-like new host onboarding
- operator role boundaries and normal fleet operation
- incident path for stale host / failed delivery / alert escalation / recovery

Exit criteria:

- the operator path can be followed without repo archaeology
- the incident path is explicit and timed
- all discovered blockers are filed as real issues

### `#187` Prepare 1.6.0 cut checklist, release notes, and go/no-go gates

Only ship when the production-readiness story is fully evidenced.

Required proof:

- external alert delivery works
- RBAC guards work
- backup and restore drill passes
- host recovery drill passes
- upgrade and rollback path is visible and repeatable
- soak evidence is committed
- buyer/operator/incident reports are committed
- final release smoke proves API, docs, public status, npm, and fleet APIs

## Acceptance Criteria

- three approved hosts remain healthy through the release proof set
- major fleet exceptions can reach an external alert target
- role boundaries are enforced for risky operations
- one backup/restore drill is documented and passes
- one host recovery flow on a fresh machine is documented and passes
- one canary rollout and one rollback are both proven
- one multi-day soak report is committed
- buyer/operator/incident evidence is committed under `reports/`
- final release smoke proves the live `1.6.0` cut
