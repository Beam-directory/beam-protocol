# Beam 1.7.0 External Dogfood Status

Status: `blocked`

Last updated: `2026-04-03`

## Summary

The repo-owned hosted-adoption work for Beam `1.7.0` is complete enough to stage a candidate, but the release is still blocked on one explicitly external gate:

- real external dogfood and adoption evidence

This is tracked as GitHub issue `#196`.

## What is already proven internally

- packaged host installer and bootstrap-token flow
- guided enrollment links and operator onboarding in the fleet UI
- operator/member management with explicit RBAC administration
- screenshot baselines and visual regression gates in CI
- fleet analytics and support-bundle export
- buyer dry run
- installer dry run
- operator dry run
- repo-owned UI smoke and visual compare

The repo now also includes a dedicated packet generator for the external pass:

- `npm run workspace:external-dogfood-pack -- --tester-name "Jane Example" --tester-email "jane@example.com" --host-label "Jane MacBook"`

That command writes one host-install packet plus one feedback template under `tmp/external-dogfood-pack/`.

## What is still missing

The missing evidence is not another local harness run. The missing evidence is a real external adoption pass that shows at least:

- one non-local host installed through the packaged bootstrap flow
- one operator using the guided enrollment path outside the dev loop
- one support bundle or fleet analytics handoff used in a realistic support/debug situation
- written feedback proving the hosted-adoption path works for someone other than the local repo maintainer

## Release implication

Beam `1.7.0` is currently:

- `candidate-ready` for the internal, repo-owned scope
- `no-go` for the public release cut

The `1.7.0` release should not be tagged until `#196` is closed with real external evidence.
