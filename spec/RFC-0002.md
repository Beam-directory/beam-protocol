## Abstract
This document specifies a federation protocol for interoperability between multiple Beam Directory instances. Federation allows organizations, public operators, and regional networks to run their own directory infrastructure while preserving global Beam agent discovery, trust evaluation, revocation exchange, and intent delivery. The protocol defines discovery, cross-directory agent resolution, registry synchronization, trust propagation, and secure routing between independent Beam Directory deployments.

## 1. Introduction & Motivation
### 1.1 Background
RFC-0001 defines the Beam Directory as the control plane for agent identity, key lookup, trust scoring, and transport discovery. That model is sufficient for early deployments, local development, and small public networks. Over time, however, a single global directory becomes operationally and politically limiting.
Large-scale agent ecosystems require regional autonomy, organizational control, legal separation, and fault isolation. The same properties that pushed electronic mail, DNS, and federated identity systems away from a single central operator also apply to Beam.
### 1.2 Why a Single Directory Does Not Scale
A single directory instance introduces several constraints:
- **Operational bottleneck**: one operator becomes responsible for uptime, scaling, and abuse handling for every participant.
- **Jurisdictional friction**: organizations may be unable or unwilling to place identity data in a foreign legal domain.
- **Trust centralization**: one party effectively controls registration policy, discovery availability, and score calculation.
- **Latency concentration**: geographically distant agents incur avoidable round trips through a central region.
- **Blast radius**: outages, key corruption, or policy mistakes affect the entire ecosystem.
Federation distributes these responsibilities across multiple directories while retaining a shared interoperability surface.
### 1.3 Sovereignty Goals
Federation exists to preserve sovereignty at multiple layers:
- **Organizational sovereignty**: an enterprise can operate its own Beam Directory for internal agents.
- **Community sovereignty**: a consortium can operate a shared directory for a sector or region.
- **Public interoperability**: public Beam directories can still exchange routing, trust, and revocation data.
- **Policy sovereignty**: each directory may apply local admission, moderation, and compliance rules.
This RFC deliberately separates **local policy** from **federation interoperability**. A directory can reject registrations or downgrade trust locally while still participating in the federation protocol.
### 1.4 Design Principles
The federated directory protocol follows these principles:
1. **Local authority first**: each agent has a home directory that is authoritative for its registration state.
2. **Pull over push**: synchronization is pull-based by default to simplify firewall traversal and operator control.
3. **Explicit trust boundaries**: remote information is informative, not automatically authoritative outside its scope.
4. **Transport reuse**: federation should reuse existing Beam identity and HTTP/TLS mechanisms where practical.
5. **Graceful partial connectivity**: directories continue operating locally even when federation links are degraded.
6. **Incremental adoption**: a single standalone directory remains valid; federation is an extension, not a prerequisite.
### 1.5 Scope
This RFC defines protocol behaviors between directories. It does not standardize:
- operator governance agreements,
- billing or settlement between directories,
- cross-directory content moderation workflows,
- mandatory global trust formulas beyond the minimum propagation rule,
- gossip-style replication or real-time event streaming.
Those topics may be covered by future RFCs.

## 2. Architecture
### 2.1 Roles
A federated Beam deployment consists of one or more Beam Directory instances. Each instance may act in one or more roles:
- **Home directory**: authoritative source for an agent's registration, active key, revocations, capabilities, and local trust inputs.
- **Peer directory**: another directory with which state is synchronized.
- **Gateway directory**: a directory that routes intents from its local agents to remote agents.
- **Public directory**: a directory that accepts registrations or lookups from the broader ecosystem.
- **Private directory**: a directory restricted to a single organization, network, or tenant boundary.
An implementation MAY support all roles within one service.
### 2.2 Authority Model
Each agent belongs to exactly one home directory. That home directory is authoritative for:
- Beam ID registration,
- public key state,
- key revocations,
- capability declarations,
- local activity-derived metadata,
- liveness information exposed for routing.
Remote directories MAY cache this data, but they MUST treat the home directory as the source of truth for the agent's identity record.
### 2.3 Topology Patterns
Beam federation supports two primary topology patterns.
#### 2.3.1 Hub-and-Spoke for Organizations
In the organizational model, one directory acts as the central hub for a company, institution, or closed ecosystem, with subordinate or edge directories syncing through it.
Example use cases:
- a multinational company with regional directories,
- a university with department-level directories,
- a managed platform with tenant-scoped edge clusters.
Properties of the hub-and-spoke model:
- policy is usually centralized at the hub,
- spokes may handle local caching and routing,
- synchronization paths are simple and auditable,
- spokes do not need full mesh connectivity.
#### 2.3.2 Mesh for Public Directories
In the public model, independent directories form a partial or full mesh of federation links.
Example use cases:
- regional public Beam registries,
- sector-specific registries such as healthcare or finance,
- open community-operated directories.
Properties of the mesh model:
- no single directory is globally privileged,
- routing paths may vary by topology,
- operators choose which peers to trust and sync from,
- revocation and registry freshness require active coordination.
### 2.4 Interoperability Baseline
To participate in federation, a directory MUST support:
- HTTPS with TLS for federation APIs,
- a stable directory base URL,
- mutual authentication between directory peers,
- pull-based registry synchronization,
- revocation synchronization,
- the trust propagation rule defined in Section 6,
- remote intent forwarding as defined in Section 8.
### 2.5 Failure Domains
Federation improves resilience by limiting failure domains:
- a local registration outage does not invalidate other directories,
- a remote sync failure does not prevent local message delivery,
- a trust or policy dispute can be isolated to specific peers,
- a directory compromise can be quarantined by disconnecting or downgrading that peer.
### 2.6 Example Deployment Shapes
A minimal enterprise deployment may include one private home directory plus one public gateway directory.
A global public deployment may include:
- `directory.us.beam.example`
- `directory.eu.beam.example`
- `directory.apac.beam.example`
Each may peer with the others while keeping separate local policy and operations.

## 3. Discovery
### 3.1 Overview
Before a directory can resolve or route to a remote agent, it must discover the responsible home directory for that agent's organization domain. Discovery is based on DNS so that organizational control maps naturally to domain ownership.
### 3.2 DNS SRV Record
A Beam-enabled organization publishes a DNS SRV record at:
```dns
_beam._tcp.example.com
```
The SRV target identifies the host and port of the Beam Directory endpoint for `example.com`.
Example:
```dns
_beam._tcp.example.com. 3600 IN SRV 10 5 443 dir1.example.com.
```
The discovering directory then constructs or retrieves the directory base URL associated with that target.
### 3.3 Directory URL Binding
This RFC assumes the SRV target resolves to a Beam Directory that exposes a canonical HTTPS base URL. Implementations MAY derive the URL as:
```text
https://dir1.example.com
```
Implementations MAY also publish a well-known metadata endpoint that confirms the canonical base URL, federation capabilities, and supported API version.
### 3.4 Discovery Process
To discover the remote directory for `agent@example.com`:
1. Extract the organization domain from the Beam identifier namespace.
2. Query DNS for `_beam._tcp.example.com`.
3. Select a target using standard SRV priority and weight rules.
4. Establish a TLS connection to the resolved directory endpoint.
5. Validate federation metadata and peer identity.
If no SRV record exists, the directory MUST treat the organization as non-federated for the purpose of this RFC.
### 3.5 Caching
Directories SHOULD cache discovery results for the DNS TTL. They MAY cache successful well-known metadata lookups separately with a local cap.
Cached discovery data MUST be invalidated when:
- the TTL expires,
- TLS validation fails for the cached target,
- federation metadata indicates a new canonical URL,
- the operator manually forces a peer refresh.
### 3.6 Multiple Endpoints
An organization MAY publish multiple SRV targets for redundancy. Directories SHOULD prefer the highest-priority healthy target and MAY fail over when the preferred endpoint is unavailable.
### 3.7 Discovery Output
Successful discovery should yield at least:
- canonical base URL,
- peer directory identifier,
- supported federation version,
- advertised sync capabilities,
- mutual-auth requirements.

## 4. Agent Resolution
### 4.1 Address Form
In federated Beam, an agent may be addressed using an organization domain form such as:
```text
agent@org.example.com
```
The local directory treats `org.example.com` as the organizational routing domain and discovers the responsible remote directory for that namespace.
### 4.2 Resolution Flow
To resolve `agent@org.example.com`, a local directory MUST:
1. Discover the remote directory for `org.example.com` using Section 3.
2. Query the remote directory for the agent record.
3. Validate the response signature or authenticated channel as required by local policy.
4. Cache the resolved record with freshness metadata.
5. Return the resulting agent metadata to the local requester or routing component.
### 4.3 Authoritative Query
The remote home directory is authoritative for whether the agent exists and for the current identity state of that agent.
The query result SHOULD include:
- canonical Beam identifier,
- display name if available,
- active public key,
- key revocation state,
- capability list,
- trust score inputs or computed trust output,
- intent transport endpoint or routing requirement,
- freshness timestamp.
### 4.4 Negative Resolution
If the remote directory returns that the agent does not exist, the local directory MAY negative-cache the result for a short duration. Negative caching SHOULD be conservative to avoid delaying newly created registrations.
### 4.5 Cache Freshness
Agent resolution caches SHOULD carry both:
- a normal time-to-live for discovery performance, and
- an invalidation path triggered by synchronized updates such as revocations or explicit tombstones.
Implementations SHOULD use short TTLs for highly dynamic fields such as presence or websocket routing hints.
### 4.6 Local Overrides
A directory MAY apply local overrides when returning a remote agent result to a local client. For example, it may:
- hide remote fields forbidden by local policy,
- downgrade locally visible trust,
- attach a warning about stale synchronization,
- block routing even if resolution succeeds.
These local overrides do not alter the authoritative remote record.
### 4.7 Resolution Failure Modes
Resolution may fail due to:
- DNS lookup failure,
- TLS failure,
- authentication failure between peers,
- remote directory timeout,
- malformed remote response,
- local policy block.
A directory SHOULD distinguish between **unknown agent**, **temporarily unreachable directory**, and **policy-blocked remote** states.

## 5. Synchronization
### 5.1 Overview
Federated Beam uses pull-based synchronization. Directories periodically fetch agent registry updates from peers rather than relying on unsolicited pushes. This keeps control with the receiving operator, works well through firewalls, and simplifies backpressure and audit logging.
### 5.2 Synchronization Unit
Synchronization exchanges agent registry state. The replicated data SHOULD include:
- agent identifiers,
- active public keys,
- revocation history,
- declared capabilities,
- trust-relevant metadata permitted for export,
- update timestamps,
- deletion or deactivation markers.
A directory MUST NOT export fields that are private by local policy unless explicitly configured to do so.
### 5.3 Pull Schedule
Directories exchange agent registries periodically. The polling cadence is a local policy decision, but implementations SHOULD support:
- frequent short polls for strategic peers,
- less frequent background polls for low-volume peers,
- immediate retry on revocation-related deltas when possible.
The schedule SHOULD adapt to peer reliability and change volume.
### 5.4 Delta Synchronization
To reduce bandwidth and replay cost, a directory SHOULD request only records changed since a known cursor, timestamp, or sequence number.
A peer SHOULD support one of these synchronization approaches:
- **time cursor**: return updates after a UTC timestamp,
- **monotonic sequence**: return updates after a committed sequence number,
- **snapshot mode**: return the full current registry when incremental state is unavailable.
### 5.5 Full Snapshots
A full snapshot is used when:
- a new federation relationship is established,
- the cursor is lost or invalid,
- the peer reports inconsistency,
- the local directory detects corruption.
Full snapshots SHOULD be paginated and checksummed.
### 5.6 Revocation Priority
Revocation data is security-critical and SHOULD be synchronized with higher urgency than ordinary profile updates. If an implementation separates update classes, revocations MUST be pulled first.
Directories SHOULD preserve original revocation timestamps from the home directory and record local receipt timestamps separately.
### 5.7 Conflict Rules
Conflicts are resolved by authority and recency:
1. The home directory wins for identity state of its own agents.
2. Within data from the same authority, the most recent valid update wins.
3. Local annotations remain local and MUST NOT overwrite remote authoritative fields.
### 5.8 Propagation Expectations
Federation is not defined as instant. Operators SHOULD size polling and retry policies to achieve timely propagation of security-sensitive changes.
For key revocations, implementations MUST account for the 24-hour propagation grace period defined in RFC-0001. During that period, directories SHOULD continue attempting sync even under degraded conditions.
### 5.9 Synchronization Integrity
Synchronization responses SHOULD be integrity-protected by the authenticated TLS channel and MAY additionally be signed at the payload level. Local implementations SHOULD log:
- peer identity,
- requested cursor,
- response size,
- applied update count,
- rejected record count,
- verification failures.
### 5.10 Deletions and Tombstones
An agent that is removed or deactivated at the home directory SHOULD be represented as a tombstone rather than disappearing silently from synchronization output. Tombstones prevent stale resurrection from caches or lagging peers.
### 5.11 Backfill and Recovery
After an outage, a directory SHOULD:
1. resume incremental sync from the last committed cursor,
2. fall back to snapshot mode if the peer rejects the cursor,
3. verify revocations before reopening routing to stale remote agents.

## 6. Trust Propagation
### 6.1 Overview
Trust in a federated system is not globally absolute. A local directory may trust a remote directory to some degree, and that remote directory may publish a trust score for one of its agents. The effective cross-directory trust must combine both judgments.
### 6.2 Minimum Rule
For cross-directory trust, the effective trust MUST be calculated as:
```text
min(local_trust, remote_directory_trust)
```
Where:
- `local_trust` is the local directory's trust level for the remote directory or federation relationship,
- `remote_directory_trust` is the trust score or trust-derived output associated with the remote agent or the remote directory's statement about that agent.
This rule ensures that a highly rated remote agent cannot exceed the confidence the local operator places in the directory that vouches for it.
### 6.3 Inputs
A local directory MAY derive `local_trust` from factors such as:
- peer onboarding verification,
- operational reliability,
- incident history,
- sector-specific compliance,
- contractual or organizational relationship,
- revocation propagation performance.
The remote value MAY derive from the home directory's own trust system.
### 6.4 Local Floors and Caps
A local directory MAY apply additional floors, caps, or policy classes after the minimum rule. For example, it may:
- cap all external agents at `0.6`,
- require `0.8` for payment intents,
- assign separate trust classes for regulated sectors.
These are local policy decisions and do not change the federation baseline formula.
### 6.5 Trust Freshness
Trust values SHOULD be timestamped. A local directory SHOULD treat stale trust inputs conservatively, especially when the peer has missed recent synchronization windows.
### 6.6 Directory Trust vs. Agent Trust
Implementations SHOULD distinguish between:
- **directory trust**: confidence in the peer operator and its data hygiene,
- **agent trust**: confidence in a specific agent based on observed behavior.
The minimum rule is intentionally simple but compatible with more nuanced internal models.
### 6.7 Examples
If:
- local trust in `directory.example.net` is `0.55`, and
- the remote directory reports `agent@example.net` with trust `0.87`,
then the effective local trust is `0.55`.
If:
- local trust in the peer is `0.91`, and
- the remote agent trust is `0.42`,
then the effective local trust is `0.42`.
### 6.8 Revocations and Trust
A revoked key or deactivated agent is not merely lower trust; it is invalid for authentication and routing according to local policy. Trust scores MUST NOT override hard security state such as revocation.

## 7. Security Considerations
### 7.1 TLS Requirement
All federation traffic MUST use TLS. Plain HTTP is not permitted for directory-to-directory communication.
TLS 1.2 or later is required. TLS 1.3 is preferred.
### 7.2 Mutual Authentication
Directories MUST mutually authenticate before exchanging synchronized registry data or routing intents. Acceptable mechanisms include:
- mutual TLS with operator-issued certificates,
- mutually authenticated Beam directory identities bound to certificates,
- another equivalent mechanism that provides strong cryptographic peer authentication.
Unauthenticated federation links MUST NOT be treated as trusted peers.
### 7.3 Peer Authorization
Authentication proves who the peer is. Authorization determines what that peer may do. A directory SHOULD explicitly authorize:
- which peers may sync,
- which peers may fetch snapshots,
- which peers may request intent forwarding,
- which namespaces or tenants a peer may query.
### 7.4 Replay and Downgrade Risks
Federation endpoints SHOULD protect against replay by using short-lived requests, nonce-bearing request signatures where applicable, and strict anti-replay controls for privileged operations.
Implementations SHOULD version negotiation explicitly to prevent protocol downgrade or ambiguous capability fallback.
### 7.5 Key and Revocation Hygiene
Because directories are identity authorities, compromise impact is high. Operators SHOULD:
- protect signing and TLS keys in hardened storage,
- maintain auditable key rotation procedures,
- prioritize revocation propagation,
- quarantine peers that fail repeated revocation validation.
### 7.6 Data Minimization
Synchronization SHOULD export only the fields necessary for interoperability. Sensitive local annotations, private usage metrics, or internal-only metadata SHOULD remain private unless there is a defined federation contract for those fields.
### 7.7 Abuse and Resource Exhaustion
Federation endpoints SHOULD enforce:
- rate limits per peer,
- pagination limits,
- bounded snapshot sizes,
- retry backoff,
- abuse monitoring for expensive resolution or sync patterns.
### 7.8 Auditability
Directories SHOULD maintain audit logs for:
- peer authentication events,
- sync requests and responses,
- revocation receipt and application,
- forwarded intent decisions,
- peer trust changes.
These logs are essential for incident response across operator boundaries.
### 7.9 Isolation on Compromise
If a peer directory is suspected compromised, a local directory SHOULD be able to:
1. suspend synchronization,
2. stop forwarding intents to that peer,
3. freeze or downgrade effective trust,
4. preserve already synced records for investigation,
5. require re-onboarding before reconnecting.

## 8. Intent Routing
### 8.1 Overview
When a local agent sends an intent to a remote agent whose home directory is elsewhere, the local directory acts as a federation gateway. It is responsible for discovering the remote directory, resolving the target, and forwarding the intent if local policy allows.
### 8.2 Local or Remote Decision
Upon receiving an intent, the directory determines whether the destination agent is local:
- if local, route using the normal Beam delivery path,
- if remote, initiate federated routing.
### 8.3 Routing Flow
The baseline remote routing flow is:
1. inspect the target Beam identifier,
2. discover the remote home directory,
3. resolve and validate the remote agent record,
4. evaluate local policy and effective trust,
5. forward the intent to the remote directory,
6. await the remote result or error,
7. relay the result back to the originating local agent.
### 8.4 Forwarding Responsibility
The local directory is not authoritative for remote delivery semantics, but it is responsible for enforcing local egress policy before forwarding. It MAY reject forwarding due to:
- insufficient effective trust,
- stale resolution data,
- blocked peer relationship,
- disallowed intent class,
- routing timeout budget.
### 8.5 Remote Directory Responsibility
The remote directory, as home or serving authority for the recipient agent, is responsible for:
- validating the forwarded intent,
- performing recipient-side routing,
- returning a result or error,
- applying local authorization and abuse controls.
### 8.6 Transparency to Agents
Where practical, routing through federation SHOULD be transparent to the originating and recipient agents. Agents continue to exchange Beam intents and results; the directories handle the cross-domain transport and discovery path.
### 8.7 Error Handling
A forwarding directory SHOULD distinguish between:
- remote agent not found,
- remote directory unreachable,
- peer authentication failure,
- remote policy rejection,
- timeout after successful forward attempt.
The originating agent SHOULD receive a Beam-compatible error result reflecting the most accurate failure class available.
### 8.8 Caching Route Decisions
A directory MAY cache remote route decisions briefly to reduce lookup overhead. It SHOULD invalidate this cache quickly when:
- the remote directory changes,
- the agent record changes,
- trust state changes materially,
- revocation data is updated.
### 8.9 Store-and-Forward
This RFC does not require durable cross-directory store-and-forward queues. A directory MAY implement queued forwarding, but if it does, it SHOULD clearly separate transient transport retries from end-to-end delivery guarantees.
### 8.10 Latency and Retries
Forwarding directories SHOULD budget for cross-directory latency and avoid indefinite waits. Retry policies SHOULD be conservative for non-idempotent intents.
### 8.11 Observability
Directories SHOULD expose metrics for:
- remote resolution latency,
- peer sync freshness,
- forwarding success rate,
- timeout rate by peer,
- rejection rate by trust or policy.
These signals help operators tune mesh connectivity and peer trust.

## 9. Open Questions
The following topics remain intentionally open in this draft:
1. a canonical well-known metadata schema for directory capability discovery,
2. whether synchronization payloads should be mandatory signed objects in addition to TLS protection,
3. how federation should represent partial capability export for privacy-sensitive agents,
4. whether remote directories should expose a live presence signal or only coarse routing readiness,
5. how multi-hop routing should be constrained in larger public meshes,
6. how operator governance and dispute handling should be standardized.
These questions do not block initial experimentation but should be addressed before a stable federation standard is published.
---
## Conclusion
Federated directories allow Beam Protocol to scale beyond a single shared control plane without giving up interoperability. By combining DNS-based discovery, home-directory authority, pull-based synchronization, bounded trust propagation, mutual authentication, and remote intent forwarding, Beam can support organizational sovereignty and public network growth at the same time.
This document is intentionally conservative. It prioritizes simple, auditable behavior over real-time complexity, leaving room for future RFCs to define richer metadata exchange, stronger settlement models, and more advanced routing topologies.
