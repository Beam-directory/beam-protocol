# RFC 0005: Beam DID Method Specification (`did:beam`)

## Abstract

This document defines the `did:beam` Decentralized Identifier method for Beam Protocol identities. The method is designed to represent Beam agents, organizations, and key-native subjects in a way that is compatible with the W3C DID Core v1.1 data model. `did:beam` uses Ed25519 keys, resolves through the Beam Directory first, and supports DNS and direct-URL fallback paths for decentralized deployment.

---

## 1. Conformance

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, **SHOULD NOT**, and **MAY** in this document are to be interpreted as described in RFC 2119 and RFC 8174.

This specification follows the W3C DID Core v1.1 model for:

- DID syntax and method-specific identifiers
- DID Documents
- verification relationships
- DID resolution results
- deactivation semantics

---

## 2. Motivation

Beam agents already use Ed25519 public keys and globally routable Beam IDs. `did:beam` adds a standards-aligned identity layer on top of those primitives so that Beam identities can be resolved, verified, and linked to verifiable credentials without depending on a single proprietary addressing scheme.

`did:beam` is intended to support:

- personal Beam identities
- organization-bound Beam agents
- key-native identifiers with no prior directory registration
- offline credential verification using embedded Ed25519 material

---

## 3. DID Syntax

### 3.1 Method Name

The method name is `beam`.

### 3.2 General Form

All identifiers use the following prefix:

```text
did:beam:<method-specific-id>
```

### 3.3 Supported Method-Specific Identifier Forms

#### 3.3.1 Personal DID

```text
did:beam:<id>
```

Where `<id>` is a Beam personal handle.

Examples:

```text
did:beam:alice
did:beam:builder_01
```

#### 3.3.2 Organization-Bound DID

```text
did:beam:<org>:<agent>
```

Where:

- `<org>` is the Beam organization slug
- `<agent>` is the Beam agent slug within that organization

Examples:

```text
did:beam:acme:assistant
did:beam:coppen:jarvis
```

#### 3.3.3 Key-Based DID

```text
did:beam:z<multibase-key>
```

Where `z<multibase-key>` is a base58btc multibase value containing an Ed25519 public key encoded with the Ed25519 multicodec prefix.

Example:

```text
did:beam:z6Mk...
```

### 3.4 Character Constraints

- Personal and organization-bound segments SHOULD use lowercase ASCII letters, digits, `_`, and `-`.
- Implementations MUST reject empty method-specific segments.
- Implementations MUST reject malformed key-based identifiers that are not valid base58btc multibase strings or that do not decode to Ed25519 multicodec bytes.

---

## 4. DID Subject Mapping

The mapping from Beam identifiers to `did:beam` is:

- `agent@org.beam.directory` → `did:beam:org:agent`
- personal handle `alice` → `did:beam:alice`
- raw Ed25519 public key → `did:beam:z<multibase-key>`

Implementations MAY expose additional aliases such as `alsoKnownAs` values pointing to Beam IDs, handles, or service URLs.

---

## 5. DID Document Model

### 5.1 Required Properties

A `did:beam` DID Document MUST contain:

- `@context`
- `id`
- `verificationMethod`
- `authentication`
- `assertionMethod`

### 5.2 Verification Method Type

`did:beam` uses `Ed25519VerificationKey2020`.

The verification method entry MUST include:

- `id`
- `type`
- `controller`
- `publicKeyMultibase`

### 5.3 Example DID Document

```json
{
  "@context": [
    "https://www.w3.org/ns/did/v1",
    "https://w3id.org/security/suites/ed25519-2020/v1"
  ],
  "id": "did:beam:acme:assistant",
  "alsoKnownAs": [
    "assistant@acme.beam.directory"
  ],
  "verificationMethod": [
    {
      "id": "did:beam:acme:assistant#key-1",
      "type": "Ed25519VerificationKey2020",
      "controller": "did:beam:acme:assistant",
      "publicKeyMultibase": "z6Mk..."
    }
  ],
  "authentication": [
    "did:beam:acme:assistant#key-1"
  ],
  "assertionMethod": [
    "did:beam:acme:assistant#key-1"
  ],
  "capabilityInvocation": [
    "did:beam:acme:assistant#key-1"
  ],
  "capabilityDelegation": [
    "did:beam:acme:assistant#key-1"
  ],
  "service": [
    {
      "id": "did:beam:acme:assistant#directory",
      "type": "BeamDirectoryService",
      "serviceEndpoint": "https://beam.directory/agents/assistant%40acme.beam.directory"
    }
  ],
  "created": "2026-03-08T12:00:00.000Z",
  "updated": "2026-03-08T12:00:00.000Z"
}
```

### 5.4 Services

`did:beam` implementations MAY publish service entries such as:

- `BeamDirectoryService`
- `DIDResolutionService`
- agent-specific messaging or discovery endpoints

Service endpoints SHOULD use HTTPS URLs.

## 6. CRUD Operations

### 6.1 Create

To create a `did:beam` DID:

1. Generate or supply an Ed25519 key pair.
2. Construct one of the valid `did:beam` identifiers.
3. Build a DID Document with at least one `Ed25519VerificationKey2020` verification method.
4. Persist the DID Document in the Beam Directory or another authoritative Beam resolver.

For Beam Directory-backed agents, registration SHOULD automatically generate and persist the DID Document.

### 6.2 Read / Resolve

To resolve a `did:beam` DID, the resolver MUST execute the algorithm in Section 7.

Successful resolution returns a DID Document with media type `application/did+json`.

### 6.3 Update

Updates MAY change:

- service endpoints
- verification relationships
- metadata such as `updated`
- active verification material during key rotation

Resolvers SHOULD treat the latest stored document as authoritative. Implementations SHOULD preserve `created` and update `updated`.

### 6.4 Deactivate

Deactivation marks the DID as no longer active.

A deactivated `did:beam` document MUST:

- retain the DID `id`
- set `deactivated: true`
- preserve sufficient metadata for verifiers to understand the terminal state

Implementations MAY retain historical documents for audit and replay protection.

---

## 7. Resolution Algorithm

Resolvers MUST attempt `did:beam` resolution in the following order:

### 7.1 Step 1 — Beam Directory API

Attempt resolution via the authoritative Beam Directory DID endpoint.

Recommended HTTP request:

```http
GET /did/{didString}
Accept: application/did+json
```

If the Beam Directory returns a valid DID Document, resolution succeeds.

### 7.2 Step 2 — DNS Fallback

If Beam Directory API resolution fails, the resolver SHOULD attempt DNS-based discovery for organization-bound identifiers.

Typical approaches include:

- discovering a Beam directory host through Beam-specific DNS metadata
- mapping an organization slug to a verified domain
- retrieving a well-known DID document location associated with that domain

If DNS discovery yields a valid authoritative DID Document, resolution succeeds.

### 7.3 Step 3 — Direct URL

If both Beam Directory API and DNS fallback fail, the resolver MAY attempt a direct URL fetch using a previously known or well-known endpoint.

Examples include:

- `https://<org-domain>/.well-known/did.json`
- `https://<org-domain>/did/{didString}`

If the fetched document is valid and its `id` matches the requested DID, resolution succeeds.

### 7.4 Failure

If all steps fail, the DID MUST be treated as unresolved.

Resolvers SHOULD distinguish between:

- malformed DID
- not found
- temporarily unavailable resolver
- document mismatch

---

## 8. Method Operations by Identifier Type

### 8.1 Personal DID

Personal DIDs are intended for standalone Beam identities and human- or agent-centric handles.

Resolvers SHOULD ensure uniqueness within the authority that issues them.

### 8.2 Organization-Bound DID

Organization-bound DIDs map cleanly to Beam Directory records and SHOULD be the default for registered Beam agents.

### 8.3 Key-Based DID

Key-based DIDs are self-certifying. Their DID Documents can be derived directly from the method-specific identifier without any registry lookup.

Resolvers MUST verify that the multibase payload decodes to an Ed25519 key before generating the DID Document.

---

## 9. DID URL Support

Implementations MAY support DID URLs with fragments such as:

```text
did:beam:acme:assistant#key-1
```

The base DID resolution algorithm applies to the DID portion. Fragment processing is performed after document retrieval.

## 10. Security Considerations

### 10.1 Key Control

Security depends on control of the Ed25519 private key. Private keys MUST be protected against disclosure and SHOULD be stored in secure key management systems.

### 10.2 Key Rotation

Rotating keys without updating the DID Document can break authentication and credential verification. Implementations SHOULD support explicit update flows and SHOULD preserve audit metadata.

### 10.3 Resolver Trust

Beam Directory resolution introduces trust in the resolver infrastructure. Clients SHOULD prefer HTTPS, validate certificates, and apply local trust policy when accepting remote DID Documents.

### 10.4 DNS Downgrade and Hijack Risks

DNS fallback introduces additional attack surface. Implementations SHOULD use verified domains, conservative caching, and, where possible, DNSSEC-aware infrastructure.

### 10.5 Direct URL Risks

Direct URL fallback MUST validate that the returned document `id` exactly matches the requested DID. Implementations SHOULD reject redirects to untrusted origins unless explicitly allowed by policy.

### 10.6 Ambiguous Personal Handles

Personal handle resolution can be ambiguous if uniqueness is not enforced. Authorities issuing personal `did:beam:<id>` identifiers SHOULD guarantee uniqueness and SHOULD reject conflicting registrations.

### 10.7 Credential Verification

Offline verification of Beam-issued credentials is only meaningful if the verifier trusts the issuer key or issuer DID. Signature validity alone does not establish business trust.

---

## 11. Privacy Considerations

- Public DID Documents SHOULD avoid embedding unnecessary personal data.
- Service endpoints SHOULD not leak internal topology unless required.
- Key-based DIDs can improve portability but may reduce unlinkability if reused across contexts.

## 13. Implementation Notes for Beam Protocol

The Beam Protocol reference implementation:

- generates DID Documents automatically at agent registration time
- stores DID Documents in a `did_documents` table
- resolves `did:beam` documents with `application/did+json`
- issues Ed25519-signed verifiable credentials for email, domain, and business verification
