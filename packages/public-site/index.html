Beam Protocol

Let one company's AI ask another company to do real work.

Beam is the shared layer for cross-company AI workflows. A request leaves your team, reaches the other company, and stays visible the whole way. You can see who handled it, what came back, and where it stalled.

One clear use case

Beam is easiest to understand when the story stays concrete.

Beam is not trying to be everything for every AI tool. The clearest starting point is one company handing work to another company with enough trust and visibility to survive real operations.

Acme asks for work

Acme sends Northwind a quote request instead of emailing, polling, and guessing what happened next.

Beam checks and routes it

Beam confirms who is talking to whom, routes the request, and records the handoff for later proof.

Northwind finishes the job

Northwind checks stock, prepares the quote, and sends the answer back through the same path.

Follow-up work stays attached

Finance can pick up the next step later without losing the thread, and Beam still shows retries or failures.

If something breaks

Beam is built so operators can see what happened.

Most cross-company automation fails quietly. Beam is designed so an operator can tell where the request stopped, what retried, and whether the other side ever answered.

Overview Check whether the system is healthy first.

Healthy runs should look like the same zero-alert, zero-dead-letter baseline shown in the hosted demo.

Intents Open one request and follow it step by step.

You can see when the handoff arrived, when it was delivered, and whether it ever finished cleanly.

Recovery See where the workflow stopped being healthy.

When delivery degrades, Beam stays explicit about retries, terminal failures, and what to do next.

  • Trace every handoff from first request to final reply.
  • Inspect alerts, audit history, rate limits, and dead letters in one place.
  • Use the same operator runbook that powers the demo and dogfood path.

What exists today

Beam already has the pieces people ask for after the first demo.

Once the use case lands, Beam already has the operator tooling, compatibility work, and demo infrastructure behind it instead of stopping at a protocol idea.

Directory and trust

Identity and policy stay explicit.

Beam keeps company identity, routing rules, and operator history visible instead of burying them in custom glue.

  • Beam IDs and DID resolution
  • Verified HTTP and WebSocket handoffs
  • Public endpoint abuse controls
  • Trace and audit APIs for operators
Durable routing

Async work does not disappear into glue code.

Retries, dedupe, restart recovery, and dead-letter handling are part of the product surface, not an afterthought.

  • Nonce dedupe and retry backoff
  • Delivery accepted vs terminal ack semantics
  • Restart recovery for in-flight work
  • Dead-letter visibility in the dashboard
SDKs and demo surface

The proof path is already operationalized.

TypeScript, Python, CLI, dashboard, docs, public site, and the seeded hosted demo already point at the same story.

  • Archived compatibility fixtures in CI
  • Hosted quickstart and dogfood scripts
  • Operator runbook and readiness reports
  • Public docs aligned to the same funnel
